# 在线语料库平台：12 课学习指南

这份指南把项目压缩为 12 课。第 1–7 课对应当前已运行的代码，第 8–12 课是下一阶段路线；规划内容不会写成“已经完成”。建议每课都按“理解 → 读代码 → 动手 → 验收”的顺序学习。

## 第 1 课：先建立可上线的工程骨架

### 本课目标

理解为什么语料平台不能从一个巨大 Django view 开始，以及 Web、PostgreSQL、Redis、Celery、文件存储各自负责什么。

### 核心认识

PostgreSQL 保存用户、权限、语料元数据和任务状态；大体量全文、Token 和索引保存在 `data/` 文件区。Redis 负责队列和缓存，Celery 负责耗时加工。这个边界避免把数百万 Token 塞进业务数据库。

### 代码地图

```text
backend/config/settings/
backend/config/urls.py
backend/apps/health/
docker-compose.local.yml
docker-compose.prod.yml
```

### 动手与验收

```powershell
docker compose -f docker-compose.local.yml up -d db redis
cd backend
.\.venv\Scripts\python manage.py check
.\.venv\Scripts\python manage.py runserver 127.0.0.1:8010
```

访问 `/healthz` 应返回 200。上线时不能沿用开发密钥、`DEBUG=true` 或任意 Host。

## 第 2 课：只读接收原始语料

### 本课目标

学会先盘点语料，再决定如何加工。阶段 1 扫描源文件、检测编码和类型、生成 manifest，但绝不移动或改写原始语料。

### 核心认识

语料处理最危险的错误不是“搜索少一条”，而是破坏唯一的原文件。manifest 把路径、大小、编码、推测语言和类型固化为可审核输入。

### 代码地图

```text
backend/apps/corpus_intake/
data/inbox/
data/manifests/
```

### 动手与验收

```powershell
.\.venv\Scripts\python manage.py scan_corpus_inbox
```

比较扫描前后的 SHA-256 与修改时间；原文件必须完全不变。manifest 不应提交到 Git，因为它可能暴露真实文件名和目录结构。

## 第 4 课：语料元数据与可见范围

### 本课目标

掌握 `Corpus`、`CorpusFile`、`CorpusDocumentation` 三类对象的职责。

### 核心认识

`Corpus` 是业务容器，`CorpusFile` 指向源文件，`CorpusDocumentation` 保存可展示统计。教师语料按等级开放；用户语料强制 `private` 且必须有 owner。

`visible_corpora_for()` 是所有列表、Documentation、KWIC、ParaConc 共用的权限事实源。测试账号当前可以看到 demo 和自己的小额上传，但看不到教师语料、其他用户上传。

### 动手与验收

分别创建两个用户的私有语料，用用户 A 访问用户 B 的 Documentation，预期 404；直接访问搜索 URL，预期 403。

## 第 5 课：统一加工流水线

### 本课目标

理解不同格式怎样进入统一的 Document、Paragraph、Sentence、Token、ParallelPair 数据契约。

### 当前 Importer

- `RawMonoTxtImporter`：中英文纯文本；
- `AlignedTsvImporter`：`zh<TAB>en` 对齐语料；
- `TaggedCorpusImporter`：已标注中英文；
- `XmlLikeImporter`：类 XML 结构；
- `PairedParagraphImporter`：保留老师提供的中英人工段落顺序，拒绝段落数量不一致的输入。
- `PairedTaggedStructureImporter`：按老师提供的 `p@n`、`s@n` 编号导入带 POS 的中英句对和段落对。

### 代码地图

```text
backend/apps/processing/contracts.py
backend/apps/processing/importers/
backend/apps/processing/artifacts.py
backend/apps/processing/services.py
backend/apps/processing/tasks.py
```

加工先写 `.staging`，成功后原子发布 `processed/<corpus_id>` 和 `indexes/<corpus_id>`。异常会回滚发布并记录失败原因。

### 中文分词与人工对齐是两件事

原始中文通过 `jieba==0.42.1` 生成词级 Token，例如“中国社会各阶级的分析”切为“中国 / 社会 / 各 / 阶级 / 的 / 分析”；英文继续使用确定性的正则 Token。带 `word/POS` 或 `token_POS` 的老师语料不再次分词，直接采用老师提供的词项和 POS。

`CorpusDocumentation.segmentation_tool` 记录每种语言的事实来源：`zh:jieba-0.42.1`、`en:regex-baseline-v1` 或 `source-provided-pos-v1`。修改分词算法必须提升索引 schema 并重建，当前版本为 1.5。

分词决定 KWIC 上下文、词频和统计单位；人工对齐决定哪段中文对应哪段英文。`PairedParagraphImporter` 只接受段落数一致的成对文件，`PairedTaggedStructureImporter` 只接受编号一致的结构；两者都不会根据分词结果猜测或重排译文。

### 动手与验收

```powershell
.\.venv\Scripts\python manage.py process_corpus --corpus-id "UUID" --sync
```

核对状态从 pending、running 到 success，Documentation 的计数与 `processing_report.json` 一致。

## 第 6 课：AntConc 风格 KWIC

### 本课目标

理解 KWIC 不是数据库模糊搜索，而是“关键词在中心、左右上下文可比较”的语料证据视图。

### 当前能力

- 中文、英文、同句短语；
- 英文忽略大小写；
- 左右窗口 0–50 Token；
- 命中总数、每页 1–100 条；
- 文件、段落、句子来源；
- L1/L2/L3/R1/R2/R3 全结果排序；
- 标注语料首词 POS 精确条件；
- 排序后分页，空上下文排在最后，原始位置作为稳定次序。

### 代码地图

```text
backend/apps/search/kwic.py
backend/apps/search/forms.py
backend/apps/search/views.py
backend/templates/search/kwic.html
```

### 关键设计

查询以 SQLite 只读模式打开 `kwic_index.sqlite`。字段、排序方向和最大短语长度都有白名单，不把用户输入拼进 SQL。元数据只读取当前页需要的 JSONL 记录。

平行语料的段内句子仍可分别进入 KWIC。中文和英文句子数量不必相同，因为译文允许拆句或合句。

## 第 7 课：ParaConc 平行语料检索

### 本课目标

理解“译文检索”的基本单位是对齐对，而不是把两种语言各搜一次再碰巧放在一起。

### 当前能力

- 中文 → 英文、英文 → 中文；
- 双语 `contains` / `not contains`；
- sentence / paragraph 对齐单元；
- 对齐顺序、分页、命中总数；
- 两列分别安全高亮；
- 对齐方法和置信度展示；
- 本人语料 TSV 流式导出，教师/demo 禁止导出。

### 代码地图

```text
backend/apps/parallel/engine.py
backend/apps/parallel/forms.py
backend/apps/parallel/views.py
backend/templates/parallel/search.html
```

### 查询逻辑

```text
alignment_unit
AND 主检索条件
AND zh_contains
AND en_contains
AND NOT zh_not_contains
AND NOT en_not_contains
ORDER BY global_position
LIMIT / OFFSET
```

旧索引必须重新加工，Web 请求不会偷偷扫描全文或改写索引。详细测试记录见 `stage-7-paraconc-and-test-upload.md`。

### 对齐粒度不能混用

老师提供的成对文件是 11 组人工段落，其中中文可分成 37 句、英文可分成 43 句。正确设计是保持 11 个段落对；句子只在各自语言内部用于 KWIC。把第 N 个中文句子与第 N 个英文句子配对，会破坏老师已经完成的工作。

另一类人工标注语料已经显式提供 `<p n>`、`<s n>` 和 POS。这类语料不需要算法对齐：平台逐项校验中英文编号，同时生成句子和段落两种视图。正文展示去除 `/n`、`_NN1` 等符号，POS 仍保留在索引中。中文 KWIC 会优先匹配标注后的词级 Token，例如“农民”；多词短语可以使用空格分隔，如“农民 运动”。

## 第 8 课：用户上传与生命周期

### 本课目标

把个人语料从浏览器安全地送入 AntConc / ParaConc 分析链路，并保证配额、并发、权限、失败恢复和删除语义在上线后仍然可靠。

### 三种上传契约

- 单语原始 TXT：选择中文或英文，加工后进入 KWIC 与 AntConc Tools；
- 中英段落人工对齐 TXT：两侧空行段落顺序必须一一对应，进入 ParaConc 段落视图；
- 中英编号/POS 人工对齐 TXT：两侧 `p@n`、`s@n` 必须一致，进入 ParaConc 句/段视图，POS 同时供 AntConc 查询。

老师和用户已经处理好的对齐关系是事实源。平台只校验并建立索引，不根据两侧分句数量重新配对。Documentation 会从只读索引展示前 5 条对齐预览。

### 生命周期与并发

```text
validate → atomic save → pending → running → success/failed → ready/retry/delete
```

测试账号单文件 2 MB、总额 5 MB；普通账号默认 30 MB。配额检查先锁定用户行，写入后再用实际字节复核。每个账号同时最多一个活动任务，这条规则既有服务检查，也有 PostgreSQL 部分唯一约束。

状态接口返回最新任务进度，页面只轮询状态；Celery 负责耗时加工。失败任务保留错误原因并允许所有者重试。删除只接受 POST，在任务结束后执行，并于数据库提交成功后清理 `user_uploads`、`processed`、`indexes` 和 `exports`。

### 安全边界

前端的文件类型提示不可信，因此服务层再次验证 `.txt`、大小、空内容和控制字节比例。原始文件名不参与磁盘路径；落盘使用 UUID、独占临时文件、`fsync` 和原子替换。用户语料固定私有，状态、重试和删除都在后端重新校验所有者。

### 验收重点

不能只测“上传成功”：还要覆盖超额、空文件、伪扩展名、二进制伪装、路径穿越、重复/并发任务、Celery 失败、跨用户访问、重试与目录清理。阶段报告 `stage-8-user-upload-lifecycle.md` 记录了 97 项回归和真实 HTTP/Celery 联调。

## 第 9 课：AntConc 统计工具

### 本课目标

掌握词频、N-Gram、搭配、目标/参照关键词、分布图和词云的统计口径，并理解为什么统计必须基于索引而不是请求期间扫描原文。

### 当前能力

- Word List：Rank、Frequency、Per million、语言/POS/文本过滤；
- N-Gram / Clusters：2–5 gram、最低频次和分页；
- Collocates：L/R span、POS、Frequency、MI、T-score、LogDice；
- Keyword：目标语料 vs 参照语料、Frequency/Range/Per million、Log-Likelihood、Chi-square、Log Ratio、正负关键词；
- Concordance Plot：每个文档归一化 100 槽；
- Wordcloud：物化词频、停用词、最小频次、TopN、18–72 px 对数缩放、三套主题、SVG 防重叠螺旋布局；
- 默认排除标点，可选择包含标点。

### 代码地图

```text
backend/apps/statistics/
backend/templates/statistics/
backend/apps/processing/artifacts.py
data/indexes/<corpus_id>/kwic_index.sqlite
  ├── word_totals
  └── word_frequencies
```

### 关键公式

```text
Per million = Frequency × 1,000,000 / Tokens
MI          = log2(Observed / Expected)
T-score     = (Observed - Expected) / sqrt(Observed)
LogDice     = 14 + log2(2 × Observed / (Node Freq + Collocate Freq))
Log-Likelihood = 2 × Σ O × ln(O / E)
Chi-square     = Σ (O - E)² / E
Log Ratio      = log2((Target Freq / Target Tokens) / (Reference Freq / Reference Tokens))
```

N-Gram 不跨句；Collocates 不跨句且排除节点自身；分布图在数据库中聚合为固定 100 槽。Keyword 动态比较两份已物化词频，不扫描全文，并按所选语言只允许 `segmentation_tool` 一致的目标/参照组合；Wordcloud 从 `word_totals` 读取，对 Frequency 做对数缩放，再用确定性椭圆螺旋和包围盒碰撞检测计算 SVG 坐标。

AntConc 的 Word/Keyword 工具统计的是索引中的词项，并不会规定中文必须按单字统计；ParaConc 的重点是双语对齐和跨语言检索，也不会替平台决定中文分词。因此，“Keyword 全是单字”应先检查平台 Token 层，不能在模板中把单字简单拼接成词。当前修复位于 `processing/text.py`，重新加工后由 Word List、Keyword、Wordcloud 和 KWIC 共用同一词级事实源。真实的单字词仍保留，这是正确统计，不是回归。

词云样式也不能靠随机 CSS 换行：随机布局会导致截图和测试不可复现。当前布局固定画布、排序、螺旋参数和调色板，相同数据会得到相同坐标；模板只负责渲染 SVG 和 KWIC 链接。详细验收见 `在线语料库平台-阶段0-10合并测试报告.md`。

## 第 10 课：CQPweb 风格复杂查询

### 本课目标

理解“解析表达式”和“执行数据库查询”为什么必须分层，并能解释复杂查询怎样安全地复用 KWIC 结果、分页和权限。

### 当前能力

- 精确词：`development`；
- 精确短语：`"high quality"`；
- 通配符：`develop*`、`?ocus`；
- 字符函数：`starts_with(develop)`、`ends_with(ment)`、`contains(velo)`；
- 属性：`[word="development"]`、`[pos="NN1"]`、`[lemma="develop"]`；
- 连续属性序列：`[pos="JJ"] [lemma="economy"]`；
- 中文与英文显式语言选择、首词 POS 快捷过滤；
- 0–50 Token 上下文、1–100 条分页和全部 L/R 排序；
- 非法表达式中文定位错误，不产生 500。

### 代码地图

```text
backend/apps/search/query_parser.py  → 字符扫描、语法验证、QueryPlan
backend/apps/search/filters.py       → TokenFilter、属性白名单、SQL 条件编译
backend/apps/search/query_engine.py  → 连续 Token JOIN、计数、排序、分页
backend/apps/search/kwic.py          → KwicHit、上下文、来源和 L/R 共同能力
backend/apps/search/forms.py         → 普通/CQP 模式与用户错误
backend/apps/search/views.py         → 权限检查和引擎选择
backend/tests/expected/search/complex_query.json
```

### 从字符串到 QueryPlan

解析器逐字符扫描，而不是用一个巨型正则“猜”整条表达式。它分别识别双引号短语、方括号属性、函数和裸词项，再把它们统一转换为：

```text
QueryPlan
└── TokenFilter[]
    ├── attribute: word | lemma | pos
    ├── operator: exact | wildcard | starts_with | ends_with | contains
    └── value: 已验证但尚未拼入 SQL 的值
```

中文裸短语会先经过与索引相同的 Jieba tokenizer，因此 `数字经济` 形成“数字 + 经济”两个连续条件。英文统一 casefold；POS 保持源文件大小写。最多 20 个 Token 条件，避免构造无限 JOIN。

未闭合引号、方括号或函数括号会报告起始位置；未知属性、未知函数、控制字符、分号、纯通配符和超长值会在访问 SQLite 前被拒绝。错误属于 `QuerySyntaxError`，表单把它转换为用户可读信息。

### 参数化 SQL，而不是字符串拼接

每个 Token 条件对应一个相邻位置 JOIN：`t1.sentence_position = t0.sentence_position + 1`。这保证短语和属性序列连续且不跨句。字段表达式来自固定映射，用户只能控制参数值：

```text
word  → tN.normalized
pos   → tN.pos
lemma → COALESCE(NULLIF(tN.lemma, ''), tN.normalized)
```

通配符只把 `*` 转换为 SQL `%`、把 `?` 转换为 `_`；用户原有的 `%`、`_` 和反斜线先转义。即使输入 `[word="x' OR 1=1 --"]`，它也只是一个普通参数，不会改变 SQL 结构。

### lemma 的真实边界

CQP 中 lemma 是独立 Token 属性。当前老师文件主要提供 word+POS，没有独立 lemma 字段，因此平台在 lemma 为空时回退到 normalized word；索引中一旦存在真实 lemma 就优先使用。这个设计让接口和查询链路完整，但不能把回退结果宣称为词形还原。未来若导入器支持 word/lemma/POS 三列，不需要修改 parser。

### 为什么复用 KWIC

`ComplexQueryEngine` 只产生与普通引擎相同的 `KwicMatch`，后续继续使用 `KwicHit` 构造、JSONL 来源读取、上下文窗口、L/R 排序和模板。权限仍在 view 首部通过 `visible_corpora_for()` 检查。这样不会产生“普通检索一套分页、复杂检索另一套分页”的双重事实源。

### 动手与验收

在《中国社会各阶级的分析》中选择 CQP 子集和中文：

1. 查询 `资产*`，应出现 18 条；
2. 查询 `contains(阶级)`，应出现 64 条；
3. 输入未闭合双引号，应显示中文错误且没有 500；
4. 将每页设为 1、排序设为 R1，翻页后确认表达式、模式和语言仍保留；
5. 回到普通 KWIC 查询“经济”，结果仍为 11 条。

在《湖南农民运动考察报告》英文侧查询 `[pos="NN1"]` 和 `[lemma="peasant"]`，再到 ParaConc 查询“第三件”，确认 Token 查询升级没有改变人工对齐。完整证据见 `在线语料库平台-阶段0-10合并测试报告.md`。

## 第 11 课：安全、审计、水印与导出（已完成）

### 目标

把现有受控导出扩展为可授权、可追踪、可限流的完整安全边界，并记录登录、检索、上传、导出和管理员操作。

### 已实现

`AuditEvent` 统一记录登录、退出、KWIC、ParaConc、AntConc 统计、上传、重试、删除、扩容、导出和管理员操作。元数据有长度和集合上限，不记录密码；IP 只采用服务端连接地址，不信任未经代理清洗的转发头。

教师语料页面按用户、语料和分钟生成 HMAC 追踪码，并在页面上重复显示动态水印。大型 KWIC/ParaConc 导出由 Celery 原子写入 `data/exports/`，限制单用户活动任务、每小时频率、结果行数、有效期和下载次数；创建与下载均校验所有者。教师/demo 语料继续只允许在线分页查看，所有批量导出返回 403。

普通用户可以申请扩大上传配额，只有启用的后台管理员可以审批。上传临时文件在最终落盘前调用扫描器接口；本地允许显式禁用，生产环境强制配置活动扫描器，内置 ClamAV INSTREAM 后端。
